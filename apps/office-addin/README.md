# @eurora/office-addin

Headless Microsoft Office add-in that bridges Microsoft Word into the Eurora desktop. The
runtime is a Shared Runtime page (no UI surface inside Word) that holds a WebSocket to the
local bridge server hosted by `crates/app/euro-browser` and answers desktop-initiated
`GET_ASSETS` / `GET_METADATA` requests with content scraped from the active document.

```
src/

‚îî‚îÄ‚îÄ word/          # Office.js wrappers (Word.run, body.text, etc.)
```

The wire types in `src/shared/bindings.ts` are generated from the Rust crate
`crates/app/euro-office`. Regenerate them whenever bridge or office types change:

```
pnpm specta:office
```

Never hand-edit `bindings.ts`.

## Manifest

`manifest.template.xml` is a token-templated Office Add-in manifest. The Tauri
installer (Phase 5) renders it at install time with absolute `file://` URLs. For
local development, `pnpm render-manifest` writes `manifest.dev.xml` pointing at
`https://localhost:3000`. Override defaults with env vars
(`OFFICE_ADDIN_DEV_HOST`, `OFFICE_ADDIN_ID`, ‚Ä¶) ‚Äî see `scripts/render-manifest.mjs`.

## Local development

```sh
# 1. Install workspace deps (from repo root)
pnpm install

# 2. Generate bindings (only needed when Rust types change)
pnpm specta:office

# 3. Start the HTTPS dev server (uses office-addin-dev-certs)
pnpm --filter @eurora/office-addin dev

# 4. Render the dev manifest and sideload it into Word
pnpm --filter @eurora/office-addin start
```

The first run of the dev server may prompt to install a local CA certificate via
`office-addin-dev-certs`; accept once and Word will trust the loopback URL.

To stop the sideloaded add-in:

```sh
pnpm --filter @eurora/office-addin stop
```

## Production build

```sh
pnpm --filter @eurora/office-addin build
```

Outputs static files to `apps/office-addin/dist/` along with `manifest.template.xml`.
The Tauri installer copies these as bundled resources (Phase 4).

## Manual smoke test

Run through these on every release candidate. Steps 1‚Äì4 are the golden
path; steps 5‚Äì7 cover the edges Phase 8's automated tests deliberately
don't reach (real Office, real OS catalog state, real reconnect).

1. **Build & sideload** the add-in as above (`pnpm --filter
@eurora/office-addin start`).
2. **Start the desktop** with `pnpm dev:desktop`.
3. **Register check** ‚Äî open any `.docx` in Word and confirm the add-in
   registers in the desktop's bridge logs as
   `app_kind=microsoft-word`. The line to look for is
   `Bridge client registered: ‚Ä¶ app_kind=Some("microsoft-word")` from
   `crates/app/euro-browser/src/server.rs`.
4. **Round-trip check** ‚Äî trigger asset generation from the desktop
   (the Word strategy fires on focus) and confirm `body.text` arrives
   on the desktop side as a `WordDocumentAsset` (no `NativeMessage`
   wrapper).
5. **Reconnect via backoff** ‚Äî kill the desktop process while Word is
   open. The add-in's WebSocket client schedules retries with
   exponential backoff capped at 30 s (see `DEFAULT_BACKOFF` in
   `src/bridge/client.ts`). Watch the runtime page's DevTools console
   for the retry log lines, then start the desktop and confirm the
   next attempt re-registers within ~30 s without sideloading again.
6. **Idempotent reinstall** ‚Äî re-run the desktop installer (or, in
   dev, call `euro_tauri::office_addin::install_for_app` from a
   debugger). Confirm:
    - **macOS:** `~/Library/Containers/com.microsoft.Word/Data/Documents/wef/com.eurora.word.xml`
      is overwritten in place; mtime advances, path doesn't change.
    - **Windows:** `%APPDATA%\Eurora\OfficeAddins\com.eurora.word.xml`
      is overwritten in place, and
      `HKCU\Software\Microsoft\Office\16.0\WEF\TrustedCatalogs\{8a4e6c52-3d0f-4a7b-9e2d-1f5c7b8e0a91}`
      keeps the same subkey GUID across runs (`reg query` it before
      and after ‚Äî `Url`, `Id`, `Flags`, `ShowInMenu` should be
      identical).
    - **Linux:** `install_for_app` returns `InstallOutcome::SkippedUnsupportedOs`
      and writes nothing. Word doesn't run natively here.
7. **Cross-platform** ‚Äî the integration must be verified on **both**
   Windows (WebView2 host) and macOS (WKWebView host) before shipping.
   Office.js loads behave subtly differently between the two
   runtimes (timing of `Office.onReady`, secure-context handling for
   `ws://127.0.0.1`); steps 3‚Äì5 must pass on each.

If the add-in fails to register on macOS the first time the desktop
launches, see also `InstallOutcome::SkippedHostNotPresent` ‚Äî Word's
sandboxed container is created on first Word launch, and the desktop
defers manifest installation rather than racing Office's own
provisioning. Launching Word once and restarting the desktop fixes it.

## Why no UI?

The first integration cut is read-only. Word has no stable "current page" concept in
Office.js, so the asset is the full `document.body.text`. All UI lives in the desktop;
the Word side is a transport. A future iteration may add a task pane with selection
context and write-back actions.
